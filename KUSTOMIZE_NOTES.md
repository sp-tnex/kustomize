# Kustomize Notes

These notes are for future reference when you want to remember what Kustomize is, why we use it, and how to run the basic commands.

## What Is Kustomize?

Kustomize is a tool used with Kubernetes YAML files.

In Kubernetes, we usually write YAML files for things like:

- Deployments
- Services
- ConfigMaps
- Secrets
- Ingress
- Namespaces

The problem starts when we need the same application in different environments.

For example:

- Development may need 1 replica.
- Production may need 10 replicas.
- Production may need CPU limits.
- Staging may use a different image tag.

Without Kustomize, we may copy the same YAML files again and again for every environment. Later, if one common thing changes, we have to update many copied files. That becomes messy very quickly.

Kustomize helps us keep one common set of Kubernetes YAML files and then customize only the parts that are different.

## Simple Meaning

Think of Kustomize like this:

```text
base YAML files + environment changes = final Kubernetes YAML
```

The base contains the common configuration.

The overlay contains changes for a specific environment like dev, staging, or prod.

Kustomize reads both and builds the final YAML for Kubernetes.

## Why Do We Use Kustomize?

We use Kustomize because it helps us:

- Avoid copying the same YAML many times
- Keep common Kubernetes config in one place
- Manage dev, staging, and production cleanly
- Change replicas, image tags, labels, namespaces, and patches per environment
- See the final generated YAML before applying it
- Use Kubernetes configuration without templates

Kustomize is also built into `kubectl`, so in many cases we do not need to install a separate tool.

## Important Idea: Kustomize Does Not Use Templates

Some tools use templates with variables, loops, and conditions.

Kustomize works differently. It starts with normal Kubernetes YAML and then applies changes on top of it.

This is useful because the base files are still valid Kubernetes files. They are easy to read and easy to apply.

## Basic Architecture

A common Kustomize project looks like this:

```text
project/
  base/
    deployment.yml
    service.yml
    kustomization.yml

  overlays/
    dev/
      kustomization.yml

    prod/
      kustomization.yml
      cpu-limit.yml
```

### Base

The `base` directory contains shared Kubernetes resources.

Example:

```text
base/
  deployment.yml
  service.yml
  kustomization.yml
```

The base should usually contain the common version of the app.

In this project, the base has:

- `base/deployment.yml`
- `base/service.yml`
- `base/kustomization.yml`

The `base/kustomization.yml` file tells Kustomize which YAML files belong to the base:

```yaml
resources:
  - deployment.yml
  - service.yml
```

### Overlay

The `overlays` directory contains environment-specific changes.

Example:

```text
overlays/prod/
  kustomization.yml
  cpu-limit.yml
```

An overlay points to the base and then changes only what it needs.

In this project, production uses:

```yaml
resources:
  - ../../base

patches:
  - path: cpu-limit.yml

replicas:
  - name: my-app
    count: 10
```

This means:

- Start with everything from `base`
- Apply the CPU limit patch
- Change the Deployment named `my-app` to 10 replicas

## What Is `kustomization.yml`?

`kustomization.yml` is the main file Kustomize looks for inside a directory.

It tells Kustomize:

- Which resources to include
- Which patches to apply
- Which image tags to change
- Which labels to add
- Which namespace to use
- Which replica count to set

Common file names are:

```text
kustomization.yml
kustomization.yaml
Kustomization
```

This repo uses `kustomization.yml`.

## Common Kustomize Fields

### `resources`

Used to include YAML files or another Kustomize directory.

```yaml
resources:
  - deployment.yml
  - service.yml
```

In an overlay:

```yaml
resources:
  - ../../base
```

### `replicas`

Used to change the number of replicas for a Deployment.

```yaml
replicas:
  - name: my-app
    count: 10
```

This changes the Deployment named `my-app` to 10 replicas.

### `patches`

Used to change part of a Kubernetes resource without copying the full file.

```yaml
patches:
  - path: cpu-limit.yml
```

The patch file only contains the part we want to change.

### `namespace`

Used to apply resources into a namespace.

```yaml
namespace: prod
```

This adds the namespace to resources generated by Kustomize.

### `namePrefix`

Adds text before resource names.

```yaml
namePrefix: prod-
```

Example:

```text
my-app becomes prod-my-app
```

### `nameSuffix`

Adds text after resource names.

```yaml
nameSuffix: -v1
```

Example:

```text
my-app becomes my-app-v1
```

### `images`

Used to change container image names or tags.

```yaml
images:
  - name: nginx
    newTag: "1.27"
```

This is useful when dev and prod use different image tags.

### `labels`

Used to add labels to generated resources.

```yaml
labels:
  - pairs:
      environment: prod
```

Labels help identify and filter resources.

## How Kustomize Builds the Final YAML

When we run Kustomize, it does not directly change the original files.

It reads the files, applies the changes in memory, and prints the final YAML.

Flow:

```text
1. Read overlay kustomization.yml
2. Find the base from resources
3. Load base YAML files
4. Apply overlay changes
5. Print or apply final YAML
```

This is why the base files remain clean.

## Useful Commands

Run these commands from the root of this repo.

### Check `kubectl`

```bash
kubectl version --client
```

### Build the base

This shows the final YAML from only the base:

```bash
kubectl kustomize base
```

Same thing using `kustomize` directly, if it is installed:

```bash
kustomize build base
```

### Build the production overlay

This shows the final YAML for production without applying it:

```bash
kubectl kustomize overlays/prod
```

Or:

```bash
kustomize build overlays/prod
```

This command is very useful because you can check what Kustomize will generate before touching the cluster.

### Apply the base to a cluster

```bash
kubectl apply -k base
```

### Apply the production overlay to a cluster

```bash
kubectl apply -k overlays/prod
```

### Delete resources created from an overlay

```bash
kubectl delete -k overlays/prod
```

### Save generated YAML to a file

```bash
kubectl kustomize overlays/prod > output.yml
```

Then you can inspect the generated file:

```bash
cat output.yml
```

### Apply generated YAML using a pipe

```bash
kubectl kustomize overlays/prod | kubectl apply -f -
```

This builds the YAML and sends it directly to `kubectl apply`.

### Compare generated YAML

```bash
kubectl diff -k overlays/prod
```

This shows what will change in the cluster before applying.

### Validate with server dry run

```bash
kubectl apply -k overlays/prod --dry-run=server
```

This asks the Kubernetes API server to check the generated resources without saving them.

### Validate with client dry run

```bash
kubectl apply -k overlays/prod --dry-run=client
```

This checks locally from the client side.

## Example From This Repo

The base Deployment is named `my-app` and has:

```yaml
replicas: 3
```

The production overlay has:

```yaml
replicas:
  - name: my-app
    count: 10
```

So when we build production:

```bash
kubectl kustomize overlays/prod
```

The final Deployment should have 10 replicas instead of 3.

The production overlay also has a patch file:

```text
overlays/prod/cpu-limit.yml
```

That patch adds a CPU limit to the `my-app` container.

## Base vs Overlay

Use `base` for things that are common everywhere.

Good examples:

- Deployment name
- Service name
- Common labels
- Container port
- Basic app structure

Use `overlay` for things that change per environment.

Good examples:

- Replica count
- Namespace
- Image tag
- Resource limits
- Environment labels
- Environment-specific patches

## Common Mistakes

### Resource name does not match

If a patch or replica change says:

```yaml
name: my-app
```

Then the resource in the base must also be named `my-app`.

If the names do not match, Kustomize cannot know what to change.

### Wrong folder path

This is common in overlays.

Example:

```yaml
resources:
  - ../../base
```

The path must correctly point from the overlay folder back to the base folder.

### Forgetting to build before apply

Before applying changes, it is a good habit to run:

```bash
kubectl kustomize overlays/prod
```

Read the output and confirm it looks correct.

### Confusing generated YAML with source YAML

Kustomize does not edit the original YAML files.

It generates final YAML from the base and overlay.

## When Should We Use Kustomize?

Use Kustomize when:

- You already have Kubernetes YAML files
- You need different environments
- Most YAML is common, but some parts are different
- You want to avoid copying full files
- You want simple Kubernetes-native configuration management

## When Kustomize May Not Be Enough

Kustomize is good for clean overlays and patches.

But for very complex packaging, conditions, loops, or advanced release management, teams may choose another tool like Helm.

That does not mean Kustomize is bad. It just means different tools solve different problems.

For learning Kubernetes configuration, Kustomize is a very good starting point.

## Quick Command Cheat Sheet

```bash
# Show final YAML from base
kubectl kustomize base

# Show final YAML from production overlay
kubectl kustomize overlays/prod

# Apply base
kubectl apply -k base

# Apply production overlay
kubectl apply -k overlays/prod

# Delete production resources
kubectl delete -k overlays/prod

# Preview changes before apply
kubectl diff -k overlays/prod

# Server-side dry run
kubectl apply -k overlays/prod --dry-run=server

# Save generated YAML
kubectl kustomize overlays/prod > output.yml
```

## Short Summary

Kustomize helps us manage Kubernetes YAML without copying the same files again and again.

The base has common resources.

The overlay has environment-specific changes.

The `kustomization.yml` file tells Kustomize how to combine everything.

Always build and check the final YAML before applying it to a real cluster.
